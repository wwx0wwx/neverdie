# 下一步实现建议

## 建议尽快补上的能力

- 把 panel 的 agent 列表持久化到本地文件或轻量数据库
- 提供 agent 注册草稿与人工批准流程
- 为节点详情增加最近若干次快照缓存和简单趋势展示
- 支持选择主磁盘和主网卡，而不是默认取第一项
- 增加 panel 到 agent 的认证机制（例如 token）
- 为 agent 增加配置文件而不只依赖环境变量

## 后续多 panel 演进方向

- 引入独立的控制面存储，用于共享 agent 配置和展示信息
- 引入 coordinator / leader 仅处理配置写入、审批与全局任务
- 让不同区域 panel 共享配置，但仍然独立拉取各自所见的实时快照
- 再往后才考虑统一域名与边缘路由策略

## 仍应继续保持 out of scope 的内容

在产品方向还未稳定前，下面这些不应过早实现：

- 全局实时遥测复制
- 分布式一致性系统
- 复杂告警编排
- 延迟 / 丢包探测矩阵
- 大规模历史时序分析平台
- 假装生产级高可用但本地无法验证的基础设施

## 本地 MVP 的验收标准

建议后续迭代都继续围绕下面的简单标准：

- 一个开发者能在一台机器上清楚地跑起来
- agent 暴露真实可读的本机数据
- panel 明确展示“这是采样快照，不是全局一致真相”
- 控制面模型与实时模型继续保持分离
